## TL;DR

Press a button, hear a voice — no install. v1.10.1 ships the **playground UI scaffold**: voice dropdown, text input, Speak button, audio element wired to the Web Audio API. The synth backend (libpiper compiled to WebAssembly + ONNX Runtime Web) lands in **v1.10.2**.

Today the Speak button POSTs to a stub endpoint that returns a sentinel marker the widget reads as "WASM build pending". The UX is real; the audio is not. The point is to lock the interaction shape before the heavy lift.

## How the widget works

| Layer | Today (v1.10.1) | v1.10.2 |
|---|---|---|
| UI | Vanilla `<select>` + `<textarea>` + `<button>` in MDX | unchanged |
| Stylesheet | External `public/playground/widget.css` referenced via `head` frontmatter | unchanged |
| Script | External `public/playground/widget.js` referenced via `head` frontmatter | unchanged |
| Endpoint | `public/api/synth.html` — static sentinel `{"status":501,"error":"not_implemented"}`; widget probes `/api/synth` first, falls back to `.html` on 404 | Astro server endpoint **or** in-browser WASM, the bare `/api/synth` probe resolves directly |
| Voice catalog | Hardcoded 4 entries (Faber, Luciana, Felipe, Amy) | Same list — actual model files lazy-load from R2 |
| Playback | Web Audio API context created on first click; `<audio>` element shown for replay | Same path; decoded PCM streams through `AudioBufferSourceNode` |
| Cold latency | n/a (no synth) | Target < 1.5 s including 63 MB Faber ONNX over residential link |

The widget treats the static HTML body as a 501 sentinel so the v1.10.1 ship-it builds cleanly on GitHub Pages (no SSR adapter, no Cloudflare Worker). When v1.10.2 lands, the only swap is the endpoint — the widget code stays.

**Why externalized in v1.10.1**: v1.9 inlined the `<script>` + `<style>` via MDX template literals. Starlight's MDX pipeline stripped or mangled them on production GitHub Pages, leaving the page with no interaction at all. v1.10.1 splits the assets out to `public/playground/` and references them via Starlight's `head` frontmatter, which always ships. See [`feedback-ship-only-tested`](https://github.com/biliboss/agent-tts/issues?q=ship-only-tested).

## Honest scope

- **No audio is produced.** Every Speak click ends in "501 — WASM build pending". This is intentional. v1.10.1 locks the interaction surface so v1.10.2 is a pure backend swap.
- **Voice list is informational.** The dropdown does not gate the request — selecting "Faber" sends `voice: "faber"` to the stub, the stub ignores it. v1.10.2 will route to the matching ONNX session.
- **No WASM compile yet.** Building libpiper for `wasm32-emscripten`, shipping the ONNX Runtime Web variant, and wiring them through `zig build` is a multi-day exercise. The scaffold ships today; the heavy lift ships next.
- **No CDN model hosting.** v1.10.2 will publish the four ONNX models on Cloudflare R2 and lazy-load on first click.
- **Browser support matrix not measured.** The widget assumes `AudioContext`, `fetch`, `URL.createObjectURL`, and `ArrayBuffer` — all baseline 2020. No polyfills.

## Build the WASM target

`scripts/build-wasm.sh` is an executable readme. It documents the four steps v1.10.2 needs:

1. Build libpiper for `wasm32-emscripten` via Emscripten SDK (`emcmake cmake … && emmake make`).
2. Ship the ONNX Runtime Web variant (`onnxruntime-web`) — the JS package, not the native `.dylib`.
3. Link the bundle via `zig build -Dtarget=wasm32-emscripten -Dwith-piper-wasm=true`.
4. Copy `zig-out/bin/agent-tts.wasm` + the ONNX runtime JS shim into `public/wasm/`.

Run it to see the steps:

```bash
./scripts/build-wasm.sh --plan
```

Today the script exits cleanly with the plan printed; v1.10.2 wires the real cmake invocations.
