## TTFA bench methodology

`agent-tts ttfa-bench --engine X --warm N` was added in v0.7.

For **piper**: bench inits the engine once (335ms cold), then runs N warm
synth+play cycles. TTFA = nanoseconds from the synth-start clock to the moment
zaudio's data source is handed to the device pump (captured via
`AudioPlayer.on_first_sample` callback that fires immediately after
`sound.start()` returns). The "first sample" is precise to one device period
(~10ms on miniaudio's macOS default config).

For **say**: bench measures wall-clock `spawn → wait` round-trip per call.
This is NOT first-sample latency — it includes total synth+playback duration
of the test string. macOS `say` runs as a child process that streams to the
audio HAL internally; there is no public hook for first-sample timing without
hijacking the audio device, which v0.7 doesn't do. **The 2229ms number is the
total playback wall time for a ~16-char Pt-BR utterance**, not the
perceptible TTFA. The real-world daemon TTFA for `say` (when the voice is
pre-warmed) is the documented sub-100ms figure from v0.2.

Net: v0.7 hits the **TTFA piper warm < 1s** roadmap target with a 10x margin
(91ms). The `say` path stays unchanged from v0.6.

## Daemon footprint with piper resident

```
$ ps -o pid,rss,vsz -p <daemon-pid>
  PID    RSS      VSZ
52557 176416 435435776
```

- RSS = 176 MB resident — dominated by ONNX runtime (~33 MB libonnxruntime
  mapped), the voice tensor allocations (~120 MB internal float buffers for
  Faber-medium), and zaudio + miniaudio pools (~4 MB). The voice ONNX file
  itself is 63 MB on disk but is loaded into the runtime's tensor format which
  multiplies the in-memory footprint.
- VSZ = 415 MB — includes the mapped onnxruntime dylib + espeak-ng data
  region. Not all resident.

For comparison, the `say`-only daemon (v0.6 minus the piper opt-in) was
~30 MB RSS. The 176 MB price is the cost of holding Piper warm; if a user
doesn't care about Piper, they skip `AGENT_TTS_PIPER=1` and avoid the load.

## Schema migration

Verified idempotent: launching v0.7 against a v0.6 queue DB (lacking the
`engine` column) adds the column via `ALTER TABLE items ADD COLUMN engine
TEXT NOT NULL DEFAULT 'say'`. Existing rows backfill to `'say'`. Re-launching
the v0.7 daemon is a no-op (PRAGMA table_info probe returns true).

The migration logic lives in `queue.zig` (`hasColumn` helper + the conditional
ALTER in `Queue.init`). v0.6 DBs in the wild auto-migrate the first time the
v0.7 daemon boots against them.

## v0.7 → v1.0 gaps

- The hidden `ttfa-bench` say path measures wall-clock playback, not
  perceived TTFA. v1.0 should either capture first-sample via `dtruss -t
  write` on the say PID (the v0.3 roadmap method) or accept the gap as
  "say's TTFA is the v0.2 daemon round-trip + ~50ms Apple internal".
- zaudio playback uses an `AudioBuffer` over the full samples slice — we
  could switch to a true streaming `decoderReadProc` so synth and playback
  overlap, dropping TTFA further. Not needed today (91ms is already well
  inside budget).
- The daemon's `runPiper` registers the daemon's own PID as `playing` so
  `agent-tts queue` shows "playing"; SKIP can't kill piper synth in
  progress (would need a synth cancellation hook in libpiper). The
  `audio_player.requestStop()` does cancel the playback half.
- Vendored zaudio carries an in-tree patch for Zig 0.16 (`std.Thread.Mutex`
  removed). Upstream is on `0.15.1`. When upstream catches up, swap to
  build.zig.zon dependency and drop the vendor/zaudio/ tree.

## Gotchas during this build

**zaudio Zig 0.16 incompatibility.** `vendor/zaudio/src/zaudio.zig` line 3160
uses `std.Thread.Mutex` (removed in 0.16). Replaced with a hand-rolled
`std.atomic.Value(bool)` spin lock — see the `AGENT-TTS PATCH (v0.7)` comment
in that file. Memory-callback contention is negligible so a spin lock is fine
in practice.

**std.Thread.sleep also removed.** `audio.zig`'s play-poll loop uses
`std.c.nanosleep` directly; we already link libc on the exe.

**zaudio.h linkLibC() API change.** Upstream's `build.zig` calls
`linkLibC()` on a `Compile` step, which 0.16 removed in favour of
`link_libc = true` on the module options. Rather than fork upstream or wait,
we wire the C sources directly into our `configureExe` and skip the upstream
`build.zig.zon` entirely. Documented in build.zig.zon's `dependencies` block.

**piper-only daemon import.** Initially the daemon imported `piper.zig`
unconditionally; this broke the no-piper build because `@cImport piper.h`
needs an include path. Fixed via a comptime-conditional `piper_mod` alias
in daemon.zig.

## What did NOT work as planned

1. **TTFA via miniaudio start-callback.** miniaudio's `notification_callback`
   gives device-level events (started, stopped) but not per-sound first-sample
   timing. The cleanest hook is to record a timestamp immediately after
   `sound.start()` returns, which is what the bench does. This is precise to
   one device period (~10ms). Acceptable for the v0.7 target; not lab-grade.

2. **Streaming via `decoderReadProc`.** Originally hoped to feed Piper chunks
   into a custom miniaudio data source so playback overlaps with synth and
   TTFA drops to "first chunk synthesized" instead of "all chunks
   synthesized". libpiper's chunk API returns variable-size buffers and
   miniaudio's data source contract expects frame counts — possible but ~50
   lines of glue that wouldn't move v0.7's TTFA materially (already 91ms
   total). Deferred to v1.0 or v1.1.

3. **zaudio upstream dependency.** Tried `zig fetch --save
   https://github.com/zig-gamedev/zaudio/archive/<commit>.tar.gz` first.
   Upstream's `build.zig` is Zig 0.15-only (`linkLibC()` method). Pivoted to
   vendor-in-tree; no breakage.

4. **espeak-ng N_PATH_HOME 160-char cap.** Documented in v0.6 baseline; the
   build under /tmp/piper-build-v07 worked first try thanks to the symlink
   workaround.
