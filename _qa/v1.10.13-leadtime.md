## Diagnosed v1.10.12 stall â€” root cause

`postfx.apply()` ran stdin write and stdout drain serially:

```zig
stdin.writeStreamingAll(io, pcm_bytes);
stdin.close(io);
// ... then ...
stdout.readerStreaming(io, &stream_buf);
while readSliceShort ...
child.wait(io);
```

When the input PCM exceeded the kernel pipe buffer (~64 KiB on macOS),
ffmpeg's output pipe filled before we started draining â†’ ffmpeg's filter
blocked on `write(stdout)` â†’ ffmpeg stopped consuming our input â†’
`writeStreamingAll(stdin)` blocked on a full input pipe. Classic two-pipe
deadlock.

The trigger in the v1.10.12 user log: `[worker] piper-ssml id=207
synth=52427ms`. A 52-second SSML synth produced ~2.3 MiB PCM (52 s Ă—
22050 Hz Ă— 2 B), well past the 64 KiB threshold. The worker thread sat
on `writeStreamingAll` forever. The queue head item (`id=210 state=playing`)
never flipped to done. Next pop returned the same head row â†’ stall.

## Fix

`src/postfx.zig::apply()` now:

1. Spawns a dedicated stdout drain `std.Thread` (concurrent with stdin write)
2. Spawns a watchdog `std.Thread` that sleeps in 50ms slices for
   `AGENT_TTS_POSTFX_TIMEOUT_MS` (default 5000 ms); on deadline it
   `SIGTERM`s the subprocess, waits 1 s grace, then `SIGKILL`
3. Both threads join before `apply()` returns; per-call arena allocations
   stay valid

Belt-and-braces: `src/daemon.zig::workerLoop` gained `defer res.queue.finishPlaying(io, item.id)`
so even if `runOne` raises an uncaught error path, the row flips from
`playing` â†’ `done` and the next loop iteration unblocks.
