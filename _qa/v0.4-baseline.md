# v0.4 — launchd auto-start (Mac Air M4)

Data: 2026-06-03
Binary: `zig-out/bin/agent-tts` (Zig 0.16, ReleaseFast, 481KB / 492584 bytes, arm64 Mach-O)
Plist label: `io.github.biliboss.agent-tts` (default) — env `AGENT_TTS_LAUNCHD_LABEL` override usado neste teste
Daemon: launchd LaunchAgent (RunAtLoad + KeepAlive on non-zero exit)

## Dry-run round-trip (test label, não toca install real)

Variável de ambiente isola o teste do install permanente:

```bash
export AGENT_TTS_LAUNCHD_LABEL=io.github.biliboss.agent-tts.test
./zig-out/bin/agent-tts daemon install
./zig-out/bin/agent-tts daemon status
./zig-out/bin/agent-tts daemon uninstall
```

3 round-trips medidos com `/usr/bin/time -p`:

| Run | install (s) | uninstall (s) |
|-----|-------------|---------------|
| 1   | 0.01        | 0.01          |
| 2   | 0.01        | 0.01          |
| 3   | 0.01        | 0.01          |

**Mediana: ~10ms install + ~10ms uninstall**. Dominado pelo fork+exec do `/bin/launchctl`. Granularidade do `/usr/bin/time` em macOS é 10ms — real ≤ 10ms (provavelmente ~5ms).

Cobre:
- `std.process.executablePath` (resolve absolute path via `_NSGetExecutablePath` no Darwin)
- `createFileAtomic` + `replace` no `~/Library/LaunchAgents/`
- `launchctl bootstrap gui/<uid> <plist>` (install) — modernizou pra evitar deprecated `launchctl load`
- `launchctl bootout gui/<uid> <plist>` (uninstall)
- `launchctl print gui/<uid>/<label>` (status, parsing-free passthrough)

## Refusal paths (defensive guards)

| Cenário | Resultado |
|---------|-----------|
| `daemon install` quando plist já existe | erro "already installed at <path> — uninstall first", exit 1 |
| `daemon uninstall` quando plist não existe | erro "not installed (no plist at <path>)", exit 1 |

Tanto install quanto uninstall recusam estado errado, evitando double-bootstrap (que o launchctl rejeita com EALREADY) e bootout em label fantasma.

## Conteúdo do plist (redacted)

`plutil -lint <plist>` → `OK`. Tamanho: 1099 bytes (host real). Conteúdo (HOME redacted):

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>io.github.biliboss.agent-tts</string>
    <key>ProgramArguments</key>
    <array>
        <string>$HOME/.../agent-tts/zig-out/bin/agent-tts</string>
        <string>daemon</string>
    </array>
    <key>RunAtLoad</key>
    <true/>
    <key>KeepAlive</key>
    <dict>
        <key>SuccessfulExit</key>
        <false/>
    </dict>
    <key>StandardOutPath</key>
    <string>$HOME/.cache/agent-tts/daemon.out.log</string>
    <key>StandardErrorPath</key>
    <string>$HOME/.cache/agent-tts/daemon.err.log</string>
    <key>WorkingDirectory</key>
    <string>$HOME/.cache/agent-tts</string>
    <key>EnvironmentVariables</key>
    <dict>
        <key>HOME</key>
        <string>$HOME</string>
    </dict>
    <key>ProcessType</key>
    <string>Background</string>
</dict>
</plist>
```

Decisões importantes:

- **KeepAlive como dict `SuccessfulExit=false`**: `<true/>` simples reinicia mesmo após shutdown limpo — `bootout` ficaria correndo atrás do próprio rabo. `SuccessfulExit=false` deixa exit 0 ser respeitado, restart só em crash.
- **HOME no EnvironmentVariables**: launchd não herda `HOME` confiavelmente em todos os contextos (pre-login, ssh, etc). Sem isso, `ipc.socketPath` resolve para `/tmp/.cache/agent-tts/sock` e o client não encontra o daemon.
- **WorkingDirectory = `~/.cache/agent-tts`**: garante que logs relativos (se algum dia o daemon escrever sem path absoluto) caiam no lugar certo.
- **ProcessType = Background**: dica pro scheduler de macOS — menor prioridade I/O, sem janela de aplicação.

## Decisões de implementação (Zig 0.16)

- **Self-locate**: `std.process.executablePath(io, &buf)`. No Darwin, vtable usa `_NSGetExecutablePath` + realpath (ver `std.Io.Threaded.processExecutablePath`). Argv[0] foi rejeitado: vem como `agent-tts` quando chamado via PATH e launchd recusa relative paths em `ProgramArguments`.
- **uid lookup**: `std.c.getuid()` (FFI direto pra libc). `std.posix` não expõe getuid como função pública em 0.16 — `std.c` é o canal padrão. Domain string: `gui/<uid>`.
- **Atomic write**: `Dir.createFileAtomic` cria ephemeral file dentro do `~/Library/LaunchAgents/` (mesmo filesystem → rename atômico), `Atomic.replace` faz o swap. launchd vê plist antigo OU novo, nunca half-written.
- **Existência do plist**: `Io.Dir.accessAbsolute(.read = true)` (note: free function, não method em 0.16 — `cwd().accessAbsolute(...)` não existe).
- **XML escape**: 5 entidades (`& < > " '`). Necessário porque alguns Macs têm HOME com espaços, e Apple aceita `&` em paths (raro, mas o test de `xmlEscape` cobre).

## v0.4 → v0.5 gaps

- Daemon que já roda quando install dispara conflita no socket — install não detecta nem mata o daemon antigo. Workaround atual: `pkill -f "agent-tts daemon"` antes de install. v0.5 pode adicionar "uninstall pre-clean".
- Sem `daemon restart` / `daemon reload` — usar `uninstall && install`.
- Sem `daemon logs` (tail dos .log). v0.5 ou v0.6 dependendo de pressão.
- Pre-warm `say -v Luciana " "` ainda roda a cada (re)start do launchd — bom (warm at boot), mas em laptop com sleep agressivo, o launchd pode matar o daemon e re-spawn várias vezes/dia. KeepAlive cobre, mas custa 340ms × N. Monitorar.

## v0.4 → SCQA

**S** (situation): v0.2 entrega daemon foreground; usuário tem que lembrar de subir manual em cada sessão. Logout/reboot derruba tudo.
**C** (complication): KPI é TTFA. Primeira chamada do dia paga cold start sempre que o usuário esqueceu o `agent-tts daemon` no terminal. Que é toda vez.
**Q** (question): Como deixar o daemon de pé sem o usuário pensar nisso?
**A** (answer): LaunchAgent plist em `~/Library/LaunchAgents/` + `launchctl bootstrap gui/<uid>` no install. Boot do Mac, login, sleep/wake — launchd ressuscita o daemon, voz Luciana já pre-warmed. TTFA quente preservado nos 0ms de round-trip do v0.2.
