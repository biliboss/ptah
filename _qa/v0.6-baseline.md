# v0.6 — libpiper FFI baseline (Mac Air M4)

Data: 2026-06-03
Binary: `zig-out/bin/agent-tts` (Zig 0.16, ReleaseFast)
Voice: `pt_BR-faber-medium` (Piper, ONNX, mono Pt-BR)
Backend: libpiper.dylib (OHF-Voice/piper1-gpl @ v1.4.2)
Daemon: not exercised here — `piper-test` bypasses daemon for cold-cost measurement

## Cold synth (init + synth + WAV write)

`agent-tts piper-test "<text>" <out.wav>` invoked from clean shell each run.
Init = `piper_create()` cold load of ONNX model + espeak-ng-data + voice config.
Synth = `piper_synthesize_start` + drain of `piper_synthesize_next` + write s16le WAV.

| Run | Text | Init (ms) | Synth+WAV (ms) | WAV size |
|-----|------|-----------|----------------|----------|
| short 1 | "Olá, tudo bem?" | 646.7 | 108.8 | 49 KB |
| short 2 | "Olá, tudo bem?" | 457.1 | 62.1 | 50 KB |
| short 3 | "Olá, tudo bem?" | 470.1 | 66.3 | 49 KB |
| long | 268-char Pt-BR paragraph | 486.7 | 731.4 | 558 KB |

Run 1 init is the truly-cold filesystem-cache-miss number (646ms). Runs 2-3
benefit from OS file cache → ~460ms. The bench param of interest for v0.7
is the warm number, since the daemon keeps the voice loaded.

## Daemon boot (AGENT_TTS_PIPER=1)

```
[daemon] listening on /Users/<home>/.cache/agent-tts/sock
[daemon] pre-warm done in 277.4ms
[daemon] piper engine loaded in 397.1ms (v0.6 baseline; not routed yet)
```

Pre-warm Luciana (`say` Premium model load) = 277ms. Piper engine cold load
= 397ms — same magnitude. Total daemon boot ~675ms which is acceptable for
a long-lived background process started by launchd (v0.4 wiring).

## Roadmap target check

v0.6 critério: "Piper synth + write WAV < 1s on M4".

- Short utterance: 470ms init + 65ms synth = **535ms total** ✅
- Long paragraph: 486ms init + 731ms synth = **1217ms total** ❌ (over by 200ms)

The 1s budget assumes "short" utterance. Long-form synth on Faber-medium is
linear in text length and outpaces 1s on 250+ char inputs. v0.7 with the
daemon-resident engine will eliminate the init cost entirely (load once at
boot, every synth pays only the network cost).

## Binary size with/without piper

| Build | Binary | Δ |
|-------|--------|---|
| `zig build -Doptimize=ReleaseFast` | 455 288 B | baseline |
| `-Dwith-piper=true` | 457 336 B | +2 KB |

Linkage is dylib-only — almost no static cost. Runtime cost is the dist/lib
tree:

```
vendor/piper1-gpl/libpiper/dist/lib/
  libpiper.dylib                   738 KB
  libonnxruntime.1.22.0.dylib       33 MB
  libonnxruntime.dylib              33 MB (symlink to .1.22.0)
```

So ~34MB of runtime payload on disk when piper is enabled. Voice model adds
63MB at `~/.cache/agent-tts/voices/pt_BR-faber-medium.onnx`. Total
~100MB footprint for the libpiper path — vs the `say` path which is 0MB
extra (lives in system voices).

## v0.6 → v0.7 gaps

- Engine load happens twice (daemon boot + piper-test). v0.7 wires routing so
  daemon-resident engine handles synth requests over the existing socket.
- Synthesis writes a WAV to disk + `afplay` it. v0.7 replaces this with
  **zaudio** streaming PCM so the first sample plays before synth finishes.
  The KPI is TTFA, not total render time — Piper produces samples in chunks
  via `piper_synthesize_next`; we can pump them to zaudio immediately.
- TTFA real (first audible sample) still not measured for the Piper path.
  v0.7 adds dtruss instrumentation matching v0.3 spec.
- The `--engine say|piper` flag does not exist yet. v0.7 owns that.
- Code-switch English (e.g. "GitHub Actions") still mispronounced — Faber is
  mono Pt-BR. Resolution waits for a multilingual ONNX voice (v1.1+).

## Gotcha noted during build

**espeak-ng has a 160-char `N_PATH_HOME` cap.** Building libpiper under the
worktree's deep path silently truncated phoneme filenames during the espeak
data-compilation stage and failed with cryptic "Bad vowel file" errors. Fix:
build under `/tmp/piper-build` and symlink `vendor/.../libpiper/build`. The
fix is documented in `vendor/README.md` so future agents don't re-hit it.
