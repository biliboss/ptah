### Sizes (with v0.7 zaudio routing landed)

| Artifact | Bytes | Notes |
|----------|------:|-------|
| `agent-tts` (host arm64) | 900 552 | ReleaseFast, libpiper OFF, zaudio ON |
| `agent-tts-universal` (fat) | 1 801 696 | x86_64 + arm64 fused via `lipo -create`, zaudio ON |

For reference: pre-v0.7 (no zaudio) the universal was ~1.08 MB and the
host arm64 ~536 KB. v0.7 zaudio + miniaudio add about +360 KB per slice
(single-header miniaudio is dense).

The universal is the arithmetic sum of the two slices plus the fat
header padding. Larger than ideal — we leave both slices in
ReleaseFast and accept ~2× size for the cross-arch convenience.

## Cross-compile

Both `aarch64-macos` and `x86_64-macos` cross-compile cleanly on this
arm64 host with Zig 0.16. No external SDK setup required beyond the
Xcode Command Line Tools (`/Library/Developer/CommandLineTools/SDKs/MacOSX.sdk`).

`build.zig` resolves the SDK location at configure time via `sdkRoot`
(probes CLT then Xcode.app) and feeds it to non-host targets as:

- library path → `libsqlite3.tbd` (multi-arch text-based dylib stub)
- system include path → `sqlite3.h`
- framework path → `CoreAudio`, `CoreFoundation`, `AudioUnit`, `AudioToolbox`
  (CoreAudio family, pulled in by vendored zaudio in v0.7).

Zig auto-resolves these for the native target only; cross-targets
would otherwise fail with `unable to find dynamic system library 'sqlite3'`
and `'sqlite3.h' not found`.

`libsqlite3.tbd` lists `x86_64-macos` + `arm64e-macos` targets; arm64
(Apple Silicon non-secure) links against the arm64e stubs without issue.

## TTFA — `say` warm

Daemon foreground (no launchd), v1.0 binary started cold and pre-warmed
Luciana. Client round-trip ACK as the conservative TTFA proxy (real
audio-device first-sample timing requires dtruss → SIP-off, not
available on this host):

| iter | round-trip (ms) | reported by client |
|------|-----------------|--------------------|
| 1 | 0.4 | `enqueued id=11 round-trip=0.4ms` |
| 2 | 0.2 | `enqueued id=12 round-trip=0.2ms` |
| 3 | 0.1 | `enqueued id=13 round-trip=0.1ms` |
| 4 | 0.1 | `enqueued id=14 round-trip=0.1ms` |
| 5 | 0.1 | `enqueued id=15 round-trip=0.1ms` |
| 6 | 0.1 | `enqueued id=16 round-trip=0.1ms` |
| 7 | 0.0 | `enqueued id=17 round-trip=0.0ms` |

**Median round-trip: 0.1ms.** Identical methodology to v0.2/v0.3
baselines — the client returns once the daemon has parsed + enqueued +
ACKed; actual `say` spawn happens server-side and is dominated by
`/usr/bin/say` fork+exec.

Pre-warm cost (boot once per daemon lifetime):

```
[daemon] pre-warm done in 275.1ms
```

For a sanity floor on the `say` fork+exec+playback path (no daemon
involved), timing bare `say -v Luciana -r 330 "x"` end-to-end:

| iter | wall-clock (ms) |
|------|-----------------|
| 1 | 800 |
| 2 | 788 |
| 3 | 791 |
| 4 | 792 |
| 5 | 792 |

That ~790ms includes spawn + model warm + playback + drain — not the
TTFA. Daemon worker thread amortizes the spawn cost across consecutive
items but still pays one fork+exec per item.

### Honest scope

- Audio-device first-sample latency (true TTFA) was NOT measured.
  `dtruss -t write` needs SIP-off; this host runs SIP-on. The 0.1ms
  round-trip is a floor for "the daemon ACKed before the audio
  started"; the real TTFA on this hardware lives somewhere between
  the pre-warm tail (~275ms) and the bare-`say` spawn (~790ms).
- Piper warm-path NOT measured. Per the v1.0 task scope, the piper
  warm metric depends on v0.7's zaudio routing landing — that work is
  in parallel and was deliberately not pulled in for this baseline.
  When v0.7 lands, `_qa/v0.7-baseline.md` should publish the warm
  number.
