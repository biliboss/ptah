**Mediana: 0.0ms** (sub-100µs). Alvo do critério v0.2 era <400ms TTFA quente. Round-trip ACK ≠ TTFA real, mas é piso confiável: daemon respondeu antes de qualquer playback começar.

Round-trip cobre:
- `connect()` no UNIX socket
- write da linha ENQUEUE
- parse no daemon + push na fila (mutex)
- write do OK
- read do OK no client

## Arquitetura realizada

```
client (one-shot)              daemon (long-lived)
  argv → Message                accept loop
  connect UNIX socket  ───────► Stream
  write ENQUEUE\t...   ───────► parse + push Queue
                                signal cond
  read OK\t<id>        ◄─────── write OK
  exit                          ▼
                                worker thread
                                pop (cond wait)
                                spawn say + wait
                                loop
```

Worker thread serializa playback — nunca dois `say` em paralelo (UX requirement).
Fila in-memory `std.ArrayList(Message)` + `std.Io.Mutex` + `std.Io.Condition`.

## v0.2 → v0.3 gaps

- Fila in-memory: não sobrevive restart (v0.3 = SQLite WAL)
- Sem `agent-tts queue` / `skip` / `clear` (v0.3)
- Daemon foreground only — usuário precisa subir manual (v0.4 = launchd)
- Pre-warm acontece uma vez no boot, mas cada `say` ainda faz fork+exec novo. Em v1.0+ avaliar embed `AVSpeechSynthesizer` se TTFA real medido (dtruss) ficar acima do alvo
- TTFA real (audio sample detection) ainda não medido — só round-trip ACK + spawn proxy
